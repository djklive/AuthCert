Le Grand Schéma de Fonctionnement de "CertifChain"
Visualisez le système comme étant composé de 3 grandes parties :
La Base de Données Centralisée (Off-Chain) : Votre PostgreSQL. Elle gère tout ce qui est dynamique, privé et relationnel : les comptes utilisateurs, les profils, les brouillons de certificats, les statistiques, les liens entre étudiants et établissements. C'est le "cerveau opérationnel".
Le Stockage de Fichiers (Off-Chain) : Un service type Amazon S3 ou un simple dossier sur votre serveur. Il stocke les fichiers statiques : les PDF des certificats, les logos des établissements, les photos de profil.
La Blockchain (On-Chain) : Votre Smart Contract sur Polygon. Elle ne stocke que le minimum vital, public et immuable : l'empreinte numérique (hash) des certificats. C'est le "notaire infalsifiable".
Maintenant, suivons le parcours de chaque utilisateur.
1. Parcours de l'Apprenant (Le Détenteur)
A. Inscription et Compte de l'Apprenant
Quand l'étudiant crée son compte qu'est ce qui se passe ? est ce que c'est un simple compte ?
Inscription : L'étudiant s'inscrit avec Nom, Prénom, Email, Mot de passe.
Création en base de données : Dans votre table users (ou apprenants), vous créez une nouvelle entrée. Vous stockez son id, ses infos, et son mot de passe hashé (très important pour la sécurité).
Création du "Portefeuille" (Wallet) : C'est l'étape blockchain cruciale. Au moment de l'inscription, votre backend va générer une paire de clés Ethereum (clé publique + clé privée) pour cet étudiant.
La Clé Publique est l'adresse du portefeuille de l'étudiant (ex: 0x123...abc). Elle est publique et sera stockée dans la table users à côté de ses autres informations. C'est son identifiant unique sur la blockchain.
La Clé Privée est le "mot de passe" secret de ce portefeuille. ELLE NE DOIT JAMAIS ÊTRE STOCKÉE EN CLAIR DANS VOTRE BASE DE DONNÉES. Vous devez la chiffrer avec une clé dérivée du mot de passe de l'utilisateur. C'est une opération complexe et sensible. Une approche plus simple pour commencer (un "compromis") est de stocker cette clé privée chiffrée dans une table séparée et hautement sécurisée (un "vault").
Le Compte : Le compte de l'étudiant est donc un simple compte utilisateur enrichi d'une identité blockchain (son adresse publique). Il se connecte avec son email/mot de passe, et votre application sait en arrière-plan quelle est son adresse on-chain.
2. Parcours de l'Établissement (L'Émetteur)
A. Inscription et Validation de l'Établissement
Quand l'etablissement cree son compte a quoi servent les cles publiques et privees ? comment son telle gere ?
Pré-inscription : L'établissement remplit un formulaire (Nom de l'école, RC, Email de contact, etc.) et soumet des justificatifs. Le compte est créé en base de données avec un statut "EN_ATTENTE_VALIDATION".
Validation par l'Admin : L'administrateur de CertifChain vérifie les documents et valide le compte. Le statut passe à "ACTIF".
Déploiement du Smart Contract : C'est l'étape magique. Une fois le compte validé, votre backend déclenche un script qui va :
Générer une nouvelle paire de clés Ethereum spécifiquement pour cet établissement. C'est le compte "propriétaire". La gestion de sa clé privée est identique à celle de l'étudiant (stockage chiffré et sécurisé).
Déployer une nouvelle instance de votre Smart Contract de certification sur la blockchain Polygon. Lors du déploiement, l'adresse du compte "propriétaire" est désignée comme owner du contrat.
L'adresse du Smart Contract nouvellement déployé (ex: 0xabc...123) est sauvegardée dans la table etablissements de votre base de données.
Conclusion : La clé privée de l'établissement sert à authentifier les actions d'écriture sur son propre smart contract. Seul le détenteur de cette clé privée peut appeler la fonction issueCertificate ou revokeCertificate. C'est la garantie cryptographique que seul l'IAI peut émettre un certificat au nom de l'IAI.
3. Le Processus de Certification (La Synergie)
Comment se passe le processus de creation de certificat et d'emission de certificat ? Que contient le certificat ? Comment est envoye le certificat a l'etudiant ?
Ce processus se fait en 5 étapes, orchestrées par l'établissement depuis son dashboard.
Étape 1 : Création des Données (Off-Chain)
L'établissement remplit un formulaire dans votre application :
Sélection de l'étudiant : Il recherche l'étudiant dans votre base de données via son email ou son nom.
Informations du diplôme : Titre ("Licence en Génie Logiciel"), Mention ("Très Bien"), Date d'obtention, etc.
Ces informations sont sauvegardées dans une table certificats de votre BDD avec un statut "BROUILLON".
Étape 2 : Génération du Document (Off-Chain)
Votre backend utilise une librairie (ex: iText pour Java, PDFKit pour Node.js) pour prendre les données de l'étape 1 et les insérer dans un modèle de PDF pré-défini (avec le logo de l'école, etc.).
Un identifiant unique (UUID) est généré pour ce certificat.
Le PDF est généré et sauvegardé dans votre stockage de fichiers (S3) sous un nom comme uuid.pdf.
Étape 3 : Création de la Preuve (Le "Hashage")
Votre backend lit le fichier PDF qui vient d'être généré et applique une fonction de hachage cryptographique dessus (SHA-256).
Résultat : Une empreinte numérique unique de 64 caractères, le hash (ex: a1b2...f9e8). C'est l'ADN du document. Ce hash est sauvegardé dans votre table certificats.
Étape 4 : Émission sur la Blockchain (On-Chain)
L'établissement clique sur "Émettre le Certificat".
Votre backend prépare une transaction à destination du Smart Contract de l'établissement :
Il récupère l'adresse du Smart Contract depuis votre BDD.
Il appelle la fonction issueCertificate du contrat en lui passant en arguments :
Le hash du certificat (a1b2...f9e8).
L'adresse publique de l'étudiant (récupérée de la table users).
D'autres métadonnées si besoin (ex: la date d'émission sous forme de timestamp).
Il signe cette transaction avec la clé privée de l'établissement. C'est la preuve d'autorité.
La transaction est envoyée à la blockchain. Une fois validée, la preuve est immuable. Le statut du certificat dans votre BDD passe à "EMIS".
Étape 5 : Notification de l'Étudiant
Une fois la transaction confirmée, votre système envoie une notification (email, notification in-app) à l'étudiant.
Le certificat apparaît maintenant sur son dashboard. Le lien se fait simplement : l'application affiche tous les certificats de la table certificats où l'etudiant_id correspond à l'ID de l'utilisateur connecté.
4. Parcours du Recruteur (La Vérification)
Comment se passe le processus de telechargement de certificat et scan de QR code ? Comment se passe la verification par le recruteur ? qu'est ce qu'il verifie le certificat json ?
Le QR Code : La Clé d'Accès
Le QR code qui est sur le PDF ne contient pas le certificat lui-même. Il contient simplement une URL vers votre application.
Cette URL pointe vers la page de vérification publique et contient l'identifiant unique du certificat.
Exemple d'URL : https://www.certifchain.cm/verify?id=uuid
Le Processus de Vérification
Scan : Le recruteur scanne le QR code avec son téléphone. Cela ouvre le navigateur sur l'URL ci-dessus.
Requête à votre Backend : Votre application reçoit la requête avec l'UUID.
Votre backend effectue alors une double-vérification en parallèle :
Vérification 1 (Off-Chain - "Qu'est-ce que je suis censé vérifier ?") : Il utilise l'UUID pour chercher le certificat dans votre base de données PostgreSQL. Il récupère toutes les informations : nom de l'étudiant, titre du diplôme, date, et surtout, le hash original qui a été stocké (a1b2...f9e8). Il récupère aussi le PDF depuis le service de stockage.
Vérification 2 (On-Chain - "La preuve est-elle valide ?") : Il se connecte à la blockchain Polygon et interroge le Smart Contract de l'établissement (dont l'adresse est dans votre BDD). Il appelle une fonction de lecture (viewCertificate(hash)) en lui passant le hash original.
La Décision :
Le Smart Contract va répondre : "Oui, ce hash a bien été enregistré pour cette adresse d'étudiant à cette date" OU "Non, je ne connais pas ce hash."
Votre backend compare les résultats. Si la blockchain confirme que le hash existe et est valide, l'authenticité est prouvée.
Affichage au Recruteur :
La page web affiche alors toutes les informations du certificat (nom, diplôme, etc.) avec un gros bandeau vert : "✅ Certificat Authentique, Émis par [Nom de l'école] et vérifié sur la blockchain."
Il n'y a pas de "certificat JSON" à vérifier pour le recruteur. L'interface lui mâche le travail et lui donne une réponse binaire (OUI/NON) basée sur cette double-vérification technique. Le recruteur voit les informations humaines (le diplôme) et la confirmation de leur authenticité.
